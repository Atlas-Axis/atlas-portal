# UUID Mapping System

## Overview

The UUID mapping system maintains bidirectional mappings between two distinct types of document identifiers in the Atlas system:

- **Notion Page UUIDs**: Generated by Notion API, tied to specific Notion pages
- **Atlas Document UUIDs**: Generated internally or externally, decoupled from Notion

This separation enables a flexible workflow where Atlas documents can exist independently of Notion, supporting markdown-first editing and external collaboration.

## Why Separate UUIDs?

### 1. Decoupling from Notion

Atlas documents remain stable even if Notion pages are recreated, moved, or changed. The Atlas UUID provides a persistent identifier independent of Notion's infrastructure.

### 2. Markdown-First Workflow

Atlas documents can exist in markdown format before Notion pages are created. The markdown file serves as the source of truth, and documents can be added to it without any Notion integration.

### 3. External Editing

External actors who don't have access to Notion can edit the Atlas markdown file and include their own UUIDs. These documents can later be synced to Notion through automated back-sync processes.

### 4. Markdown-to-Notion Sync

When new documents are added to the Atlas markdown file, the back-sync automation creates corresponding Notion pages and establishes UUID mappings. The Atlas UUID remains stable while a new Notion page UUID is generated.

## UUID Types

### Notion Page UUIDs

- **Source**: Generated by Notion when pages are created, accessible through Notion API
- **Format**: Standard UUID (e.g., `1b2f2ff0-8d73-8095-bb8f-ed052141f936`)
- **Scope**: Tied to specific Notion database pages
- **Stability**: Changes if page is deleted and recreated
- **Field Name**: `notion_page_id`

### Atlas Document UUIDs

- **Source**: Generated internally (during Notion import) or externally (in Atlas markdown file)
- **Format**: Any valid UUID version (v4, v7, etc.)
- **Scope**: Independent of Notion
- **Stability**: Persistent across Notion page recreations
- **Field Name**: `atlas_document_uuid` or `uuid`

## Architecture

### Storage

UUID mappings are stored in the `uuid_mapping` database table:

```sql
CREATE TABLE uuid_mapping (
  notion_page_id UUID NOT NULL UNIQUE,
  atlas_document_uuid UUID NOT NULL UNIQUE
);
```

**Key Features:**

- Bidirectional uniqueness constraints (one-to-one mapping)
- No foreign key constraints (allows mappings to exist before/after documents)

### Creation

UUID mappings are created in two scenarios:

#### 1. Automatic Creation During Notion Import

When importing Notion databases to Supabase (`insert-pages-in-batches.ts`):

1. New Atlas documents are inserted into `notion_database_pages`
2. For each new document, a new UUID is generated
3. A mapping entry is created: `(notion_page_id, atlas_document_uuid)`
4. Process is batched (500 mappings per batch) for performance

#### 2. Manual Generation for Existing Documents

The `scripts/generate-uuid-mapping.ts` script generates mappings for existing documents that may be missing a UUID mapping pair:

1. Loads all existing mappings from `uuid_mapping`
2. Queries all current Notion pages from `notion_database_pages_current`
3. Identifies pages without mappings
4. Generates UUID for each missing page
5. Inserts new mappings in batches

It should never happen that a Notion page UUID that exists in Supabase has no Atlas document UUID pair, but this script exists just in case.

### Loading

The `loadUuidMappings()` function provides efficient bidirectional lookup:

```typescript
interface UuidMappings {
  notionPageIDsToAtlasUUIDs: Map<string, string>;
  atlasUUIDsToNotionPageIds: Map<string, string>;
}
```

**Process:**

1. Queries `uuid_mapping` table in pages (1000 rows per page)
2. Populates both directional maps simultaneously
3. Returns O(1) lookup maps for both directions

**Serialization Support:**

For embedding in HTML/JSON contexts, mappings can be converted to plain objects:

- `serializeUuidMappings()`: Converts Maps to Record objects
- `deserializeUuidMappings()`: Converts Record objects back to Maps

## Usage Patterns

### 1. Tree Building and Document Numbering

**Location**: `atlas-tree-builder.ts`

**Purpose**: Generate maps from Atlas UUIDs to document metadata

**Process:**

1. Load UUID mappings at tree construction time
2. During traversal, look up Atlas UUID from Notion page ID
3. Create maps: `atlasUUIDsToGeneratedDocNumbers` and `atlasUUIDsToDocNames`
4. These maps are used for mention updates and exports

### 2. Rich Text Mention Updates

**Location**: `atlas-tree-mentions.ts` (called from `atlas-tree-builder.ts` Step 8)

**Purpose**: Update outdated document numbers in Notion mention objects by matching the Notion page UUID

**Process:**

1. Extract Notion page ID from mention (`mention.page.id`)
2. Look up Atlas UUID using `notionPageIDsToAtlasUUIDs`
3. Look up current document number from `atlasUUIDsToGeneratedDocNumbers`
4. Look up current document name from `atlasUUIDsToDocNames`
5. Update mention `plain_text` to `"{number} - {name}"`

**Example Transformation:**

```json
// Before
{
  "type": "mention",
  "plain_text": "A.2.4",  // Outdated!
  "mention": {
    "page": { "id": "1b2f2ff0-..." }
  }
}

// After
{
  "type": "mention",
  "plain_text": "A.1.3 - General Provisions",  // Corrected!
  "mention": {
    "page": { "id": "1b2f2ff0-..." }
  }
}
```

### 3. Markdown Conversion

#### Notion Rich Text → Markdown (`rich-text-to-markdown.ts`)

**Purpose**: Convert Notion links to Atlas UUID links in markdown

**Process:**

1. Detect Notion URLs in rich text links (`https://www.notion.so/...`)
2. Extract Notion page ID from URL
3. Look up Atlas UUID using `notionPageIDsToAtlasUUIDs`
4. Replace Notion URL with Atlas UUID in markdown link

**Example:**

```markdown
<!-- Notion link -->

[Document](https://www.notion.so/page-id-123)

<!-- Converted to Atlas UUID -->

[Document](atlas-uuid-abc-456)
```

#### Markdown → Notion Rich Text (`markdown-to-rich-text.ts`)

**Purpose**: Convert Atlas UUID links back to Notion URLs

**Process:**

1. Detect Atlas UUID in markdown link
2. Look up Notion page ID using `atlasUUIDsToNotionPageIds`
3. Convert to Notion URL format
4. Create Notion mention object with page ID

### 4. JSON Export

**Location**: `atlas-node-tree-to-standardized-atlas-node-tree.ts`

**Purpose**: Include Atlas UUIDs in standardized JSON export

**Process:**

1. For each tree node, look up Atlas UUID using `notionPageIDsToAtlasUUIDs`
2. Include Atlas UUID in `StandardizedAtlasDocument.uuid` field
3. Enables markdown export with stable UUIDs

### 5. UI Rendering

**Locations**: `sidebar.tsx`, `content-tree.tsx`, `search-modal.tsx`

**Purpose**: Generate React keys and Notion links

**Process:**

1. Load UUID mappings on page load
2. For each document node, look up Notion page ID using `atlasUUIDsToNotionPageIds`
3. Use Notion page ID for:
   - Stable React component keys
   - Links to Notion pages
   - Document identification

### 6. Notion Sync (Markdown to Notion)

**Location**: `notion-property-builder.ts`

**Purpose**: Build Notion property values with proper UUID handling

**Process:**

1. Convert markdown content to Notion rich text
2. During conversion, use UUID mappings to:
   - Convert Atlas UUIDs in links to Notion page IDs
   - Create proper Notion mention objects
   - Build relationship properties

## Data Structures and UUID Types

### Database Tables

#### `notion_database_pages`

- **UUID Field**: `notion_page_id` (Notion page UUID)
- **Purpose**: Store Notion pages imported from Notion API
- **Relationships**: Child ID arrays use Notion page UUIDs

#### `uuid_mapping`

- **UUID Fields**:
  - `notion_page_id` (Notion page UUID)
  - `atlas_document_uuid` (Atlas document UUID)
- **Purpose**: Bidirectional mapping between UUID types

### Atlas Markdown Format

In Atlas markdown files, UUIDs are Atlas document UUIDs:

```markdown
# A.1.2 - Document Name [Section] <!-- UUID: 305e2bd6-a594-4aec-8713-adbe7bc87120 -->

Content with [link to document](atlas-uuid-abc-456).
```

## Operations

### Creating Mappings for New Documents

**Scenario**: Importing new documents from Notion

**Workflow:**

1. `import-notion-databases.ts` fetches pages from Notion API
2. `upsertPagesInBatches()` inserts pages into `notion_database_pages`
3. For `changeType === 'insert'`, `insertUuidMappingsForBatch()` is called
4. UUID v4 is generated for each new page
5. Mappings are inserted into `uuid_mapping` table

### Loading Mappings for Operations

**Scenario**: Building Atlas tree or exporting markdown

**Workflow:**

1. Call `loadUuidMappings()` at start of operation
2. Receive bidirectional Maps for O(1) lookups
3. Pass `UuidMappings` to functions that need UUID conversion
4. Use appropriate map direction based on conversion needs

### Converting Between UUID Types

**Notion UUID → Atlas UUID:**

```typescript
const atlasUUID = uuidMappings.notionPageIDsToAtlasUUIDs.get(notionPageId);
```

**Atlas UUID → Notion UUID:**

```typescript
const notionPageId = uuidMappings.atlasUUIDsToNotionPageIds.get(atlasUUID);
```

### Handling Missing Mappings

**When Atlas UUID is not found:**

- Log warning with context (document type, Notion page ID)
- Use placeholder like `"[Unknown]"` in user-facing output
- Continue processing other documents

**When Notion page ID is not found:**

- Log warning about missing reverse mapping
- May indicate document only exists in markdown (not yet synced to Notion)
- Can still export markdown with Atlas UUID

## Best Practices

### 1. Always Load Mappings Once

Load UUID mappings once at the start of an operation and pass them through the call stack. Avoid reloading mappings repeatedly.

### 2. Handle Missing Mappings Gracefully

Not all documents have mappings (new markdown documents, deleted Notion pages). Always check for `null`/`undefined` and provide fallbacks.

### 3. Use Correct UUID Type in Queries

Be explicit about which UUID type you're querying:

- Database queries use Notion page UUIDs
- Markdown/JSON export uses Atlas document UUIDs

### 4. Preserve Atlas UUIDs in Markdown

When editing Atlas markdown files, never modify UUIDs in comment tags. They provide stable identity across sync operations.

### 5. Generate UUIDs Appropriately

- **Internal generation**: Use any UUID generation library (v4, v7, etc.)
- **External generation**: Allow any valid UUID format from external editors
- **Validation**: Ensure UUIDs are unique within the Atlas system
