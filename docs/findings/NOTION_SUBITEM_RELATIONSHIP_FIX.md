# Notion Sub-item Relationship Fix

## Overview

This document describes a known Notion issue where bidirectional relationships in the Agent Scope Database can become desynchronized, and how the system handles it. This information is preserved for future reference.

> **Status (December 9, 2025):** The affected data has been cleaned up in both Notion and Supabase. The workarounds described below remain in place as defensive measures against future occurrences.

## The Issue

**Known Notion behavior:**

1. The Agent Scope Database has "Parent item" / "Sub-item" configured as a **bidirectional relationship** (`dual_property` type in Notion API)

2. Under certain conditions, the relationship can become desynchronized:
   - Child page's "Parent item" property: **Correctly set** to the parent's page ID
   - Parent page's "Sub-item" property: **Empty** (missing the child)

3. This desynchronization exists **in Notion itself**, not in Supabase. Verified via direct Notion API queries:

   ```
   Parent "Sub-item" property: 0 relations
   Child "Parent item" property: 1 relation (pointing to parent)
   ```

4. **Historical note:** As of December 8, 2025, approximately 4,500+ pages in the Agent Scope Database were affected in local development Test Notion databases (generated by `create-test-notion-databases.ts`). This was cleaned up on December 9, 2025.

5. The tree builder in `atlas-tree-builder.ts` has a workaround that uses `parent_notion_page_id` to reconstruct parent-child relationships during import, compensating for the missing `child_agent_scope_ids` data

## Impact (If Issue Recurs)

When the Sub-item property is empty on parent pages:

- The `child_agent_scope_ids` array in Supabase would be empty for affected parents
- Without the tree builder workaround, these children would appear as "orphaned nodes"
- The Markdown→Notion diff would incorrectly detect these as "new" pages, creating duplicates

## How It's Handled

### 1. Tree Builder Workaround (Import)

The `atlas-tree-builder.ts` file contains a "second pass" in `createLookupMaps()` that handles this issue at runtime:

```typescript
// Second pass: For internally nested databases (Agent Scope Database, Sections & Primary Docs),
// also use parent_notion_page_id to build relationships. This handles cases where Notion's
// bidirectional relations are not synced.
```

This workaround:

- Iterates through all pages with `parent_notion_page_id` set
- Adds them to their parent's `child_agent_scope_ids` (or `child_section_and_primary_doc_ids`) array
- Logs how many children were added: `📊 Added X children from parent_notion_page_id`
